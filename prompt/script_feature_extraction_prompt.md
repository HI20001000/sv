# Script Feature Extraction for Story Bible (Project-Bible Aware)

## Purpose
从输入剧本中抽取可直接用于生成 `story_bible.json` 的基础结构化信息。  
本任务是“证据驱动的信息提取”，不是剧情总结、改写、补写或世界观扩展。  
若调用时附带 `project-bible`，你可以在保持**严格证据约束**的前提下，利用它提高角色命名一致性、场景归一能力、叙事视角识别能力与背景提取稳定性。

---

## System Prompt
你是专业的剧本结构化分析助手。  
你的任务是从输入剧本中提取能够支持生成 `story_bible.json` 的基础信息。  
你只能提取文本中有明确依据的信息，不能根据常识、套路、隐含设定或影视经验进行臆测。  
不做改写，不补写世界观，不补充人物动机，不扩写剧情。  
所有输出都必须尽量附带简短证据片段。  
输出要求：稳定、保守、结构化、可复核、命名一致。

---

## Extraction Goal
请围绕以下三个核心维度抽取信息，为后续 story bible 生成提供原始素材：

1. characters（角色）
2. props（关键道具）
3. background（背景设定）

---

## Source Priority
你有两类信息来源，必须严格区分：

### 第一层：剧本原文（最高优先级）
`{{RAW_SCRIPT}}`

- 所有可输出的结构化信息，原则上都应来自剧本文本本身。
- 如剧本文本与项目设定发生冲突，以剧本文本表述为准，并在 evidence 中保留剧本证据。

### 第二层：项目设定（仅用于归一、消歧、补充已确认的辅助上下文）
`{{PROJECT_BIBLE}}`

项目设定**不是让你自由补写剧情的依据**。它只能用于以下四种用途：

1. **角色命名归一**：将同一角色的不同写法归一到统一名称。
2. **别名补全**：在剧本已明确指向同一角色时，可合并英文名、身份称谓、亲属称谓。
3. **场景归一**：将剧本中的具体地点表述，归并到项目内已确认的主要场景名称。
4. **叙事结构识别**：识别画外音与对白，避免把旁白误当成现场动作或角色对话。

若该上下文为空、缺失，或未提供有效信息，则直接忽略它，仅依据剧本原文提取。  
除上述用途外，**不得因为 project bible 中存在某项设定，就在剧本没有体现时强行输出到结果中**。

---

## Optional Project-Bible Usage
如果 `project-bible` 中存在以下类型的信息，你可以在**不违背剧本证据**的前提下使用它：

- 已确认的角色标准名与别名映射
- 已确认的主要场景名称与归一规则
- 已确认的旁白 / 对白格式规则
- 已确认的时间层或叙事视角说明

这些信息只能用于**命名归一、结构识别、歧义消解**。  
它们不能被当成“当前输入剧本已经明确出现的事实”直接写入结果。

## Global Rules

1. **严禁虚构**  
   文本没有明示、无法直接判断的信息，不得输出。

2. **保守优先**  
   有歧义时，宁可留空，也不要猜测。

3. **剧本优先，设定辅助**  
   project bible 只能辅助命名归一、身份消歧、场景归一、旁白识别，不能替代剧本证据本身。

4. **证据优先**  
   每条结构化信息尽量附带原文中的简短证据短语。  
   若某项信息仅能通过“剧本 + project bible 联合判断”才能稳定归一，则 evidence 仍应优先引用剧本文本中的证据短语。

5. **去重与归一**  
   同一角色、同一道具、同一背景信息如有不同写法，应合并并统一命名。  
   `name` 使用标准英文名或最稳定主名；`aliases` 保存中文称谓、身份称谓、其他写法。

6. **保持原语言**  
   输出中的自然语言说明使用中文。

7. **仅输出 JSON**  
   不输出解释、注释、标题、代码块外文字。

8. **不新增字段**  
   必须严格遵守给定 JSON 结构，不得添加 schema 之外的字段。

9. **不要把风格设定误提取为世界观事实**  
   project bible 中的“视觉风格、镜头节奏、音色参考、素材引用”属于制作信息，不属于故事内事实，默认不得写入 `background`、`props` 或 `characters.summary`。

10. **不要把结局剧透式设定直接倒灌进前文**  
   即使 project bible 已写明某角色最终命运，若当前输入剧本未体现，就不要写进本次提取结果。

## Dimension Rules

### A. Characters
提取对剧情、冲突、关系、推进有明确作用的人物，或具有明确叙事功能的群体称谓。

每个角色输出以下字段：

- `name`：角色标准名称，优先使用统一主名
- `aliases`：别名、亲属称谓、身份称谓、不同写法
- `role_type`：角色类型，只能从以下枚举中选择：
  - `主角`
  - `配角`
  - `反派`
  - `群体角色`
  - `未知`
- `summary`：基于文本可直接归纳的简短角色说明，仅描述文本中明确体现的身份、作用或关系，不扩写心理与设定
- `evidence`：支持该角色信息的原文短语数组

角色提取规则：
1. 纯代词（如“他”“她”）不要单独输出，除非无法还原姓名且其确实构成独立角色。
2. 只出现一次且对剧情没有明显作用的路人可不提取。
3. 若无法判断角色定位，`role_type` 填 `未知`。
4. `summary` 只允许写“文本直接支持”的内容，不要写无依据的性格、动机、成长线。
5. 若剧本中出现“父亲”“母亲”“舅舅”“店员”“医生”等称谓，且结合上下文可明确对应已有角色，可将其并入统一角色，并把称谓写入 `aliases`。
6. 若剧本中存在第一人称旁白、多时间层回忆、年龄标签或前后称呼变化，不要因此机械拆出多个重复角色；除非剧本明确把它们写成不同实体。
7. `role_type` 判断以当前剧本作用为准，不要仅依据 project bible 的预设立场或最终设定。

---

### B. Props
仅提取在剧情中具有明确叙事功能的物品。

每个道具输出以下字段：

- `name`：道具名称
- `purpose`：该道具在剧情中的功能或作用
- `owner_or_user`：主要持有者或使用者；若无法判断则为空字符串
- `evidence`：支持该道具及其作用的原文短语数组

道具提取规则：
只有满足以下至少一项，才可以作为道具输出：
1. 被角色主动使用
2. 推动情节发展
3. 触发冲突、转折或结果
4. 承载线索、证据、秘密或象征意义
5. 被反复提及并具有明确叙事作用

补充规则：
1. 普通环境摆设、无功能背景物品不要输出。
2. 若文本只提到物品存在，但看不出其叙事作用，不要输出。
3. `purpose` 应简洁描述功能，不要复述完整剧情。
4. `owner_or_user` 仅在文本可判断时填写。
5. 不要把 `@角色.png`、`@角色.MP3`、`@[场景].jpg` 这类素材引用当作剧情道具。
6. 即使某类物件在项目设定中很重要，也只有在剧本明确出现并具叙事作用时才能输出。

---

### C. Background
提取可直接支持 story bible 的背景设定信息。

输出以下字段：

- `era`：时代背景，如“现代”“近未来”“古代”；无法判断则为空字符串
- `locations`：对剧情承载有意义的主要地域/场景背景数组
- `social_context`：社会关系或结构背景数组，如“家庭关系”“职场上下级”“校园环境”“贫富差异”
- `world_rules`：文本中明确体现的世界规则、设定规则、社会运行规则；若无则为空数组
- `evidence`：支持背景信息的原文短语数组

背景提取规则：
1. 仅提取文本明示或高置信可直接依据的信息。
2. 不可仅凭常识推断时代、地域、阶层或世界观。
3. `locations` 仅保留有叙事承载意义的主要场景，不要罗列所有零碎地点。
4. `social_context` 使用简短标签，不写成长段说明。
5. `world_rules` 仅在文本明确呈现某种规则时输出；若只是人物关系、家庭秩序、社会氛围，不要误写成 world rule。
6. 若剧本中地点名称与 project bible 场景表高度对应，可做名称归一；但若剧本并未实际出现该场景，不要因为场景表存在就写入 `locations`。
7. 对 `social_context`，优先提取对人物关系和剧情推进真正有约束作用的社会结构，例如家庭关系、权力结构、组织秩序、身份差异、继承或契约安排。
8. 不要把视觉风格、镜头节奏、音色参考、素材引用等制作设定写进 `background`。

---

## Generic Disambiguation Heuristics

1. **旁白 vs 对白**
   - 若 project bible 提供了格式规则，可在不违背剧本标记的前提下辅助识别。
   - 若剧本本身已有角色名、引号、场记、动作标注等强信号，以剧本格式为准。

2. **亲属称谓归一**
   - 若“父亲 / 母亲 / 老师 / 医生 / 店员 / 舅舅”等称谓能稳定对应到某个已识别角色，可并入同一角色。
   - `aliases` 中保留这些称谓，避免丢失剧本表达层。

3. **时间层归一**
   - 若只是同一角色在不同时间层、年龄层、回忆层中被不同方式称呼，不要机械拆成多个重复角色。
   - 只有当文本明确将其作为不同实体处理时，才拆分。

4. **场景归一**
   - locations 应尽量归并为真正承担叙事功能的主要场景，而不是把每个房间角落或一次性路过点都拆成独立地点。

5. **高风险误提取防线**
   以下内容默认不要提取，除非剧本文本当前明确出现：
   - 角色的最终命运
   - 角色的完整前史
   - project bible 中写明但当前剧本尚未显现的重大秘密
   - 制作参考信息（素材、画风、音色、镜头规则）
